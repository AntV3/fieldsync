# FieldSync Audit — 2026-04-19

Branch: `claude/audit-fieldsync-VqdEF`

A whole-system audit of FieldSync (React 19 + Vite + Supabase + Capacitor;
~277 source files / ~90k LOC; 39 canonical Supabase migrations + 58 legacy
SQL files in `database/`; 9 test files / 311 tests passing).

This report consolidates four parallel reviews:
1. Build / lint / test / dependency health
2. Security (auth, RLS, secrets, XSS, deps, mobile)
3. Code quality and architecture
4. Database schema, RLS policies, migrations

Findings are grouped by severity. Each item names a concrete file (and line
where useful) so it can be triaged into issues directly.

## 2. Security

22 findings (4 critical, 5 high, 5 medium, 8 informational).

### CRITICAL

**SEC-C1 — Permissive `USING (true)` policies still in code path.**
`database/schema.sql`, `database/add_pin.sql:19-20,33,37`,
`database/migration_photo_reliability.sql`, plus 16 instances across
`supabase/migrations/` (notably `20250116_fix_ambiguous_project_id_v2.sql`
lines 64-68 and `20260315_trade_profiles.sql` line 103+). Multi-tenant
isolation is enforced *only* by the application + SECURITY DEFINER
functions. If `field_sessions` validation logic is broken or bypassed,
all rows are anon-readable. Replace with policies that join on
`user_companies` or call `has_valid_field_session()`; treat
`USING (true)` as a lint failure in CI.

**SEC-C2 — PIN stored without hashing (assumed plaintext).**
`database/add_pin.sql:5` adds a `pin` column with no `pgcrypto`
treatment, and `database/fix_pin_validation_case_sensitivity.sql:29-59`
compares with `TRIM(p_pin)`. A DB read exposes every project PIN.
Hash with `crypt(pin, gen_salt('bf', 12))` and compare via `crypt()`.

**SEC-C3 — Demo-mode auth bypasses password.**
`src/lib/supabase.js:94-108` — when `isSupabaseConfigured` is false the
sign-in path matches by email only. `src/lib/supabaseClient.js:19-25`
correctly hard-fails in production builds, but the dev-only path is
still risky if a developer ships a build with env vars missing.
Recommend: gate the demo branch on `import.meta.env.DEV` rather than on
`isSupabaseConfigured`, and surface a banner.

**SEC-C4 — Field-session token surface.**
`src/lib/fieldSession.js:76-100`. Token (32 hex bytes) lives in
`sessionStorage`; passed as custom `x-field-session` header. Issues:
(a) no client-side expiry pre-check, (b) no rotation on privilege
change, (c) token is only as strong as `validate_pin_and_create_session`
RPC entropy. Pair with SEC-H1.

### HIGH

**SEC-H1 — `SECURITY DEFINER` functions without `SET search_path`.**
50+ SECURITY DEFINER functions in `supabase/migrations/20260218_*`,
`20260219_*`, `20260301_security_hardening.sql`,
`20260309_audit_hardening.sql`, `20260310_*`. Only ~11 `search_path`
mentions across the whole tree. Postgres advisory: every
SECURITY DEFINER function should `SET search_path = public, pg_temp` to
prevent shadow-schema attacks.

**SEC-H2 — File uploads have no server-side MIME / size validation.**
`src/lib/db/documentOps.js`, plus storage policies in
`database/migration_field_photo_uploads.sql` and
`supabase/migrations/20260218_fix_photo_bucket.sql`,
`20260310_launch_security.sql:66-89`. Mime type comes from `file.type`
(client supplied). Risk: SVG XSS, oversized uploads, path-style
filename attacks. Enforce a whitelist + size cap in a Storage trigger
or Edge Function; sanitize names via `src/lib/sanitize.js:112-127`
(which already exists and is good).

**SEC-H3 — Cross-tenant signature visibility.**
`database/migration_signatures_anon_access.sql:12-22,48-51`. Anon
INSERT/SELECT on signatures gates only on signature_request existence,
not on the request's company_id. A foreman who knows a foreign
`signature_request_id` can read or sign it. Add company_id check.

**SEC-H4 — Public token entropy & one-time use.**
`src/lib/db/financialOps.js:328-373` — share tokens are 12 chars from
62-char alphabet (~71 bits, marginal). Signature tokens come from the
`generate_signature_token()` RPC — confirm it uses ≥256 bits. Public
endpoints `/sign/:token` and `/view/:token` (`vercel.json:4-9`) have no
rate limiting, no one-time-use, and end up in browser history /
Referer. Increase to 24 chars and add per-IP rate limits at the edge.

**SEC-H5 — CSRF module not actually wired in.**
`src/lib/csrf.js` is a competent implementation but no production
fetcher uses `createSecureFetch`. Either wire it through the Supabase
client headers (and validate in an Edge Function) or delete to remove
the false sense of security.

### MEDIUM

- **SEC-M1** Observability logs may carry PII (`src/lib/observability.js:171-206`). Confirm RLS on `error_log` / `query_metrics` (`20241228_observability_tables.sql`) restricts to service_role.
- **SEC-M2** Foreman name & onboarding state in `localStorage` (`src/lib/localStorageHelpers.js`). Move to `sessionStorage` or drop.
- **SEC-M3** TOTP secret displayed plaintext in `src/components/MFASetup.jsx`. Show QR by default, mask the seed.
- **SEC-M4** Public `/sign` and `/view` URLs lack one-time-use semantics (see SEC-H4).
- **SEC-M5** Duplicate `20250116_fix_ambiguous_project_id.sql` and `…_v2.sql` both define overlapping policies; collapse to one canonical file.

### INFO

- **SEC-I1** Capacitor config (`capacitor.config.json:36-40`) is good — no mixed content, debugging disabled.
- **SEC-I2** CSP (`vercel.json:42`) is solid; consider dropping `'unsafe-inline'` from `style-src` (use nonces). `script-src 'self'` is correct.
- **SEC-I3** `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` in client bundle — by Supabase design, OK; double-check `.env*` files are gitignored.
- **SEC-I4** Invitation tokens used directly in `getInvitationByToken` — Supabase parameterizes, low risk; add format validation for defense in depth.
- **SEC-I5** Auth rate limiting RPCs exist (`migration_security_hardening.sql:30-71`) and are called from `fix_pin_validation_case_sensitivity.sql:33`. Good.

## 3. Database

### Migration system

Two parallel directories:
- `database/` — 58 legacy flat files (assume already applied to prod)
- `supabase/migrations/` — 39 timestamped files, **canonical** going forward
- `supabase/rollbacks/` — only 1 of 39 has a paired rollback

Recommendations:
1. Mark `database/` as historical in a README; new work only in `supabase/migrations/`.
2. Backfill rollbacks for at least every migration that touches RLS / GRANTs / table shape since 2026-02-18.
3. Replace `apply_all_pending_migrations.sql` (a giant manual catch-up file) with the regular `supabase db push` flow.

### RLS posture

23 tables have RLS enabled. Patterns observed:
- **Field tables** (daily_reports, disposal_loads, t_and_m_tickets, crew_checkins, injury_reports, material_requests, punch_list_items, disposal_truck_counts) grant SELECT/INSERT/UPDATE/(DELETE) directly to `anon` and rely on `field_sessions` validation in SECURITY DEFINER functions. This is the entire trust boundary — see SEC-C1, SEC-H1.
- **`trade_templates`** is `USING (true)` for everyone (`20260315_trade_profiles.sql:103+`). Trade workflows leak to competitors. Gate to authenticated company members.
- **`field_sessions`** itself is correctly inaccessible except via DEFINER functions.
- **Signatures** anon access (SEC-H3).

### Performance

`supabase/migrations/20250118_performance_indexes.sql` adds reasonable
indexes on the high-traffic field tables. Gaps:
- `disposal_truck_counts` (added 20260306) has no documented index.
- `custom_field_data(entity_type, project_id)` — likely needed.
- Consider partial indexes for `status='open'` filters.

### Schema smells

- Status columns started as `TEXT`; only got CHECK constraints in `20260309_audit_hardening.sql:13-37`. Audit remaining tables for the same retrofit.
- ENUMs (`disposal_load_type`) require migrations to extend (`20260306_add_disposal_load_types.sql`). A `TEXT` + CHECK + lookup table would be more flexible.
- No soft-delete strategy. Hard deletes via FK `CASCADE` could clobber audit trails for daily reports / disposal loads — confirm against retention requirements.
- Denormalized money fields (computed at insert) can drift if rates later change. Add a recalculation script or store the rate alongside the amount.
- JSONB fields in `trade_templates` have no schema check; add a CHECK that uses `jsonb_typeof(...)` at minimum.

### Storage

`tm-photos` bucket policies (`20260218_fix_photo_bucket.sql`,
`20260310_launch_security.sql:66-89`) extract `projectId` via
`storage.foldername(name)[2]`. Path-confusion risk if a malformed
upload key sneaks in. Reject names that don't match `^uuid/uuid/.*` in
the upload path validator.

## 4. Code Quality & Architecture

### High-impact issues

**ARCH-H1 — God components.** Top offenders:
- `src/components/TMList.jsx` — 1,568 LOC
- `src/components/TMForm.jsx` — 1,432 LOC
- `src/components/Dashboard.jsx` — 1,360 LOC (manages 58+ `useState` calls)
- `src/components/SignaturePage.jsx` — 1,320 LOC
- `src/components/AppEntry.jsx` — 1,251 LOC
- `src/components/cor/CORForm.jsx` — 1,193 LOC

Split each into per-tab / per-modal subcomponents and lift queries into hooks.

**ARCH-H2 — Monolithic data-access modules.**
- `src/lib/corOps.js` — 2,514 LOC
- `src/lib/db/projectOps.js` — 1,574 LOC
- `src/lib/db/fieldOps.js` — 1,333 LOC
- `src/lib/db/tmOps.js` — 1,011 LOC

The `supabase.js` barrel + `db/*Ops.js` pattern is good in spirit; the
implementation is just oversized. Split per resource verb (read / write /
subscribe).

**ARCH-H3 — Direct DB access from components.** ~345 occurrences of
`db.*` or `supabase.from(...)` invoked from `.jsx` files (e.g.
`Dashboard.jsx:2`, `AppEntry.jsx`). Schema changes ripple outward.
Funnel through hooks (`useAsyncData`, `useProjectEdit`) or services.

**ARCH-H4 — Silent error swallowing.** 6 `.catch(() => {})` instances:
`src/components/OfflineIndicator.jsx`, `src/components/Dashboard.jsx`,
`src/lib/offlineManager.js` (×3), `src/lib/corExportPipeline.js`. At
minimum log via the existing `src/lib/observability.js` layer.

**ARCH-H5 — No `React.memo` on hot list rows.** 241 useCallback/useMemo
calls but zero `React.memo`. List components (`TMList`,
`InjuryReportsList`, etc.) re-render their entire row tree on every
parent state change.

### Medium

- **ARCH-M1** 409 `console.*` calls leak into the production bundle. Replace with `observe.*` from `src/lib/observability.js` and add an ESLint `no-console` rule.
- **ARCH-M2** Seven calculation modules totaling ~3,200 LOC (`cashFlowCalculations.js`, `forecastCalculations.js`, `riskCalculations.js`, `qualityCalculations.js`, `resourceCalculations.js`, `corCalculations.js`, `earnedValueCalculations.js`) duplicate base patterns (sum, average, trend). Extract a `metrics/base.js`.
- **ARCH-M3** Zero type safety — no TypeScript, inconsistent JSDoc. Start by enforcing JSDoc on `src/lib/db/*` exports.
- **ARCH-M4** State management — `Dashboard.jsx` has 58+ `useState`s + a `projectDetailsCacheRef` with an arbitrary 30s TTL. Either move shared data into React Query / SWR (with proper invalidation) or a tiny store (Zustand).
- **ARCH-M5** **Test coverage** — only 9 test files / 311 tests for 277 source files. No tests for any `db/*Ops.js`, hooks, or components. Calc modules and `sanitize` are well-covered; nothing else is. Wire `vitest --coverage` into CI with a floor.

### Low

- **ARCH-L1** Stale planning docs at repo root (`APPLY_FIX.md`, `SECURITY_AUDIT.md`, `SECURITY_VULNERABILITIES.md`, `ENHANCEMENT_PLAN.md`, `IMPROVEMENT_PLAN.md`, `AUDIT-UI-UX.md`, `CODE_REVIEW.md`). Move to `docs/audits/` with dates so it's clear what's current.
- **ARCH-L2** 315 `aria-*` attributes but no axe-core or keyboard-nav tests. Add `vitest-axe`.
- **ARCH-L3** Bundle budget — add `vite-plugin-compression` and a CI size check. Lazy-load `xlsx` and `jspdf` at action time (they're chunked but still pulled by the import graph in some pages).

### Strengths

- `src/lib/errorHandler.js` and `src/lib/observability.js` are well-designed and underused — wire them everywhere.
- `db/*Ops.js` barrel split is the right shape, just needs further breakup.
- Hooks (`useAsyncData`, `useAuthState`, `usePortfolioMetrics`) are well-documented with proper dep arrays.
- Offline-first via `offlineManager.js` is a real differentiator.
- Code-splitting in `vite.config.js` already isolates the heaviest libs.
- CSP, HSTS, X-Frame-Options, Permissions-Policy in `vercel.json` are correctly set.
